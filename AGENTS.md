# Regras Para IAs Neste Projeto

Este projeto e o dono das extracoes, tabelas, views, indices e migrations do ETL.

- Cada macaco no seu galho: mudancas de schema, regra materializada ou view do ETL ficam neste repositorio. Nao coloque migrations do ETL no projeto `dashboards`.
- O projeto `dashboards` deve consumir o contrato publicado pelo ETL; se o contrato precisar mudar, altere primeiro aqui e deixe a mudanca explicita em migration, script base e view.
- Sempre que criar ou alterar uma migration em `database/migrations`, atualize tambem os scripts canonicos correspondentes em `database/tabelas`, `database/views`, `database/indices` e `database/validacao`, quando aplicavel. Uma recriacao limpa do banco precisa nascer ja atualizada.
- Arquitetura e separacao de responsabilidades: mantenha extracoes, transformacoes, banco, validacoes e contratos publicados nos diretorios corretos do ETL. Nao coloque regra do Dashboard aqui, nao misture camada de apresentacao com carga de dados e nao crie atalhos fora da arquitetura existente.
- Mesmo quando a IA operar e validar somente em DEV, mantenha os artefatos consumidos por producao prontos para o proximo start/deploy humano: migrations, scripts canonicos, views, indices, exemplos e contratos publicados devem ficar sincronizados com a mudanca. Nao execute rotinas de producao, mas tambem nao deixe o caminho de producao defasado.
- Arquivos sensiveis para commit/push: sempre que criar ou alterar classe, arquivo, config, dump, planilha, log, credencial, segredo, token, certificado ou qualquer artefato que nao deve ir para o GitHub, atualize o `.gitignore` correspondente para excluir esse caminho antes de commitar. Se o arquivo sensivel ja estiver versionado, avise no resumo e nao o exponha em diff ou commit.
- Regras caras de BI que dependem de normalizacao, data de referencia ou elegibilidade devem ser materializadas nas tabelas durante a carga/backfill do ETL sempre que forem usadas por dashboards recorrentes.
- CRUD e exclusoes: dados de negocio extraidos, cadastros, tabelas de apoio e dimensoes governadas nao devem ser apagados fisicamente por comportamento de tela ou rotina comum. Quando algo precisar "sumir" para o usuario, use inativacao/arquivamento/logica temporal (`ativo=false`, `archived`, `deleted_at`, vigencia ou equivalente) e mantenha o registro para backup, auditoria, historico e integridade dos paineis.
- Hard delete/truncate em dado de negocio so pode ocorrer com pedido explicito do operador humano, justificativa clara, backup, migration controlada e teste/validacao. Limpezas tecnicas de staging temporario, tabelas `#temp`, cache descartavel ou rotinas de retencao documentadas nao entram nessa regra, mas devem ser escopadas e seguras.
- Relacionamentos com entidades que podem ser inativadas devem preservar o historico com `LEFT JOIN` ou fallback equivalente; views publicadas para o Dashboard devem retornar valores seguros como `SEM_MAP`, "Usuario Inativo" ou outro fallback definido, em vez de quebrar a consulta.
- Antes de implementar uma mudanca, leia o contrato existente e crie/ajuste teste que reproduza a regra, bug ou fluxo esperado sempre que isso for viavel. Implemente, rode os testes focados e depois a validacao relevante do ETL (`src/test`, scripts de validacao SQL, migrations e consultas de sanidade). Nao considere concluido sem verificar que funcionou.
- Ao finalizar qualquer mudanca, confira se atributos, aliases de SQL, nomes de colunas, schemas, payloads, contratos publicados e consumidores esperados batem entre si. Nao deixe campo inventado, classe sem uso, import morto, duplicacao desnecessaria ou codigo fora do padrao de clean code e engenharia de software do projeto.
- Use as pastas de testes e validacao do projeto para proteger comportamento. Se uma mudanca nao tiver teste automatizado possivel, registre no resumo qual validacao manual/local foi feita.
- Encoding e mojibake: preserve acentos, simbolos e aliases exatamente como publicados pelo banco/API. Nao aceite texto corrompido por encoding, como UTF-8 lido como ANSI/Windows-1252; se encontrar alias, migration, view, CSV, seed, fixture, doc ou teste com caracteres estranhos no lugar de acentos/simbolos, corrija a origem e valide usando UTF-8 antes de seguir.
- Por padrao, nao execute rotinas de producao, cargas contra banco de producao, deploys, scripts de start de producao ou testes em servicos/dominios publicos.
- Excecao controlada: a IA so pode subir, reiniciar, parar, liberar portas, executar rotinas/testes de producao ou acessar servicos/dominios de producao quando o operador humano escrever literalmente `pode subir pra produção` na mensagem atual. Pedidos sem essa frase literal, como "suba agora", "teste em prod" ou "pode testar", nao autorizam producao.
- Quando houver a autorizacao literal `pode subir pra produção`, execute somente o necessario para o teste/deploy solicitado, registre o que foi feito e pare ao concluir a validacao. Nao mantenha alteracoes, processos ou rotinas extras alem do escopo pedido.
